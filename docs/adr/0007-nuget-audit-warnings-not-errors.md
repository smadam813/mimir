# NuGet advisories warn loudly but never fail the build

The NuGet audit codes — `NU1900` (failure to fetch advisory data from the feed) and `NU1901`–`NU1904` (vulnerability advisories, by severity) — are carved out of `TreatWarningsAsErrors` via `WarningsNotAsErrors` in `Directory.Build.props`. They surface as warnings on every restore and build; every other warning in the solution still escalates to an error. Audit itself stays at its .NET 10 defaults: enabled, full transitive depth (`NuGetAuditMode` = `all`), all severities reported. Decided while closing issue [#44](https://github.com/smadam813/mimir/issues/44), the advisory half of the two-way split with [#30](https://github.com/smadam813/mimir/issues/30)'s deterministic restore.

**The failure mode this removes**: `NuGetAudit` runs on every restore against live advisory data from nuget.org. A security advisory published against an *unchanged* — typically transitive — dependency raises `NU1902`/`NU1903` on the next restore, which `TreatWarningsAsErrors` turned into a build error: a previously green commit went red on a day nobody touched the repository. The lock files shipped by [#30](https://github.com/smadam813/mimir/issues/30) do not help, because `packages.lock.json` pins *which* packages resolve while audit evaluates that locked graph against *today's* advisories. Determinism and audit are orthogonal levers; this ADR is the audit one. `NU1900` is the same hazard through a different door: it fires when the advisory data itself cannot be *fetched* — a nuget.org blip or rate-limit during a CI restore — so escalating it fails the build on network weather rather than on anything in the repository.

**`NU1603` is deliberately not carved out**, although issue [#44](https://github.com/smadam813/mimir/issues/44) originally grouped it with the audit codes as "the same time-dependent class". That grouping does not survive contact with the docs: `NU1603` (package resolved higher than requested) also fires for a *permanent authoring error* — a version pin that does not exist on the feed, silently resolving to something higher — which reproduces deterministically on every restore of an unchanged graph and should fail, loudly, on the PR that introduces it. (For a *direct* reference the same defect surfaces as sibling code `NU1601` — verified empirically here: a nonexistent `12.0.99` pin resolved to `13.0.1` and failed the build as an error. Neither sibling is carved out.) The genuinely time-dependent variant requires a previously resolving version to vanish from the feed, and nuget.org — this repo's only source — does not allow package deletion; CI additionally restores in locked mode against committed lock files. On this repo, `NU1603`'s realistic trigger is a defect in the change being built, not feed drift, so it keeps escalating.

**The carve-out is a hand-enumerated list, and that is a known limitation**: `WarningsNotAsErrors` takes no wildcards, so a future SDK that introduces a new audit code would escalate it until the list is extended. This stays inside the deliberate-change boundary the ADR defends: `global.json` pins a full SDK version and CI installs exactly that, so a new code can only start firing after a deliberate SDK-bump commit — and it fails *that* PR, whose author extends the list in the same change. (`rollForward` can surface a newer SDK on local machines earlier; a local-only warning-turned-error on an SDK the repo hasn't adopted yet is noisy but harmless.) There is no CI check for list drift because the toolchain publishes no machine-readable set of audit codes to check against.

**Two acceptance criteria pulled against each other**: a new advisory must not redden a green commit without a deliberate change, and the repo must keep zero warning *suppressions*. `WarningsNotAsErrors` is the only lever that satisfies both at once. It is not a suppression: the advisories still print on every restore, in local builds and in CI logs, with their codes and severities intact — `NoWarn` would erase them. What changes is only the escalation to a hard failure.

**Considered options**:

- **`NuGetAuditMode` = `direct`** — audit only direct dependencies. This stops the observed hazard (the reddening advisories are overwhelmingly transitive) but makes the build fully silent on transitive vulnerabilities, the largest signal loss of any option. Rejected.
- **`NuGetAuditLevel`** (`low`/`moderate`/`high`/`critical`) — escalate only at or above a severity threshold. Narrows by severity rather than depth, but a newly published advisory at or above the threshold still reddens a green commit, so it fails the first criterion on its own. Rejected as a primary lever; it also composes poorly here, since with the carve-out in place there is no escalation left for it to threshold.
- **`NuGetAuditSuppress` per advisory** — NuGet's native mechanism for acknowledging an advisory without failing the build. Disqualified twice over: it addresses advisories by URL, so it can only name advisories that already exist and cannot prevent the redden from one published tomorrow; and it removes the acknowledged advisory from the output entirely, which is a suppression. Rejected.
- **`NoWarn` on the NU codes** — hides the advisories entirely, in violation of the second criterion. Rejected outright.

**What is lost, plainly**: CI never hard-fails on a known vulnerability. An advisory against this dependency graph is only actioned when a human or agent *reads the warnings* — in the local build output or the CI log — and decides to bump the affected package. There is no mechanism in this decision that forces that reading; a warning nobody looks at is functionally a suppression. Accepted because the alternative — time-dependent hard failures on unchanged commits — punishes whoever happens to push next rather than whoever owns the dependency, and because a single-maintainer repo with agents reading build output has a realistic chance of actually seeing the warnings. If that stops being true, the correcting move is a scheduled audit job (e.g. a cron workflow running `dotnet list package --vulnerable` that opens an issue), which restores an active signal without re-coupling advisories to unrelated commits — tracked as [#50](https://github.com/smadam813/mimir/issues/50) so the option is not forgotten.

**Constraint carried forward for update automation**: if Dependabot/Renovate is ever introduced to act on these warnings, its configuration must group `Npgsql.EntityFrameworkCore.PostgreSQL` and `Microsoft.EntityFrameworkCore.Design` into a single update — the lockstep documented in `Directory.Packages.props` — so it never bumps one without the other.
