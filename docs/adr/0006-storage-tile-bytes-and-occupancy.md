# Storage tile reports bytes and occupancy, not row counts

The §8 Storage tile reports each table's on-disk size plus a three-valued occupancy (`Populated` / `Empty` / `Unknown`), and never a row count. Size comes from the catalog and is always exact; occupancy comes from `EXISTS`, which reads live tuples through MVCC and so cannot disagree with a count in the same snapshot. Neither is a statistic. Decided while making the tile affordable under §10's keep-forever retention, where an exact `count(*)` every 15 seconds is an unbounded sequential scan over the largest tables in the database.

**Both cheap estimators were measured lying, in opposite directions, on PostgreSQL 17.10.** A table created empty, analyzed while empty, then filled with 200,000 rows reports `reltuples = 0`, `relpages = 0` and `n_live_tup = 0` — a populated table shown as empty, which is exactly the shape an EF migration produces. The mirror case: 200,000 rows inserted, analyzed, then fully deleted reports `reltuples = 200,000` and `n_live_tup = 50,000` for a table holding nothing. That second case fires precisely when a user is checking whether a §8.2 hard Delete of sensitive Events took effect, which is the worst possible moment to be wrong. Choosing an estimator is choosing which way to lie, so no design that reads one was viable — including a size-gated hybrid, which still has to estimate above the gate.

**Considered options**: estimates everywhere with a `~` label (fails the honesty constraint above); exact counts gated by table size, estimating above the threshold (same failure above the gate, plus a threshold knob to tune); a slower probe cadence (reduces duty cycle linearly, changes correctness not at all); exact counts on a user-triggered detail surface (still open, and additive to this contract rather than a replacement for it).

**What is lost, plainly**: exact cardinality leaves the health strip. "How many Episodes do I have?" is no longer answerable at a glance, and trend-watching becomes 1.2 GB → 1.4 GB rather than 40k → 55k events. This is coarsest in early use, when a handful of Episodes will not visibly move the byte figure. Accepted because §10 states the tile's purpose as making growth visible under keep-forever, which is a bytes question, and because an absent number serves that better than a wrong one. Per-run item throughput stays on the Distillation and Harvester tiles, so "is the pipeline moving" remains visible in the strip.

**Do not justify this on buffer-cache eviction.** That argument was raised and falsified: a warmed table held one buffer before and after three full 355 MB `count(*)` scans, because Postgres uses a 256 KB `BAS_BULKREAD` ring for scans of tables larger than `shared_buffers / 4`. A large sequential scan does not flood the pool at the expense of the §7 recall path's working set. The justification is honesty and cost, nothing else — recorded here so the decision is not reopened on a false premise.

`EXISTS` is cheap rather than strictly bounded, and the distinction is worth stating. It stops at the first live tuple, so on a populated table it is effectively constant (measured: 0.015 ms, 2 buffers, on a million rows). Its bad case is a table with no live rows but heavy dead-tuple bloat, where it must scan the dead tuples to conclude "empty" — but that is exactly the case a `count(*)` also pays in full, so this is never worse than what it replaces, and autovacuum resolves it.

One bug was fixed in passing, independent of the estimator question: `pg_tables` returns partitioned parents *and* their children, so the previous query discovered a partitioned table twice and summed it to double its real size (measured: 50,000 rows reported as 100,000). Discovery now reads `pg_class` with `NOT relispartition` and rolls leaf sizes up into the parent.

**The probe deliberately runs without a wrapping transaction, and that took two attempts to get right.** An early draft wrapped discovery and occupancy in `REPEATABLE READ` so their answers would "describe one moment". That reasoning was wrong in three separate ways, each measured on PG 17.10:

- **It does not make sizes and occupancy agree.** `pg_total_relation_size` reads the filesystem and ignores the snapshot: inside one `REPEATABLE READ` transaction the same table measured 8 KB and then 27 MB. `EXISTS` *does* honour the snapshot. So the transaction produced "27 MB · empty" for a table holding 50,000 rows — a populated table shown as empty, the one thing this tile must never say. Statement-level snapshots make each answer as fresh as it can be, and the occupancy legs are a single statement, so they are already atomic with respect to each other.
- **It does not stop a concurrent `DROP`.** Below `SERIALIZABLE`, PostgreSQL resolves relation names against a fresh catalog snapshot, so `EXISTS` on a dropped table still raises `42P01` inside the transaction. That is caught, and the probe reports occupancy as `Unknown` for the round rather than guessing.
- **It actively created a crash.** A dropped table stays visible in the transaction's `pg_class` snapshot while `pg_total_relation_size` returns **NULL** for it, which threw an `InvalidCastException` that no `DbException` filter could catch and flipped the tile to `Degraded` with "Postgres unavailable" while the database was perfectly healthy. Without the transaction, the dropped table is simply not listed.

Discovery still skips NULL-sized rows as defence in depth, and the `42P01` catch remains. A startup migration makes all of this routine rather than exotic.
