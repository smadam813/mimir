# The scheduled audit fails loudly when it cannot fetch advisory data

A weekly GitHub Actions workflow (`.github/workflows/security-audit.yml`, Mondays 06:17 UTC plus `workflow_dispatch`) re-runs the build's advisory audit decoupled from the commit cadence: locked-mode restore against the committed `packages.lock.json` files, then `dotnet list Mimir.slnx package --vulnerable --include-transitive --format json` — the same advisory source (nuget.org) and the same dependency graph the build's `NuGetAudit` evaluates, so the job and the build agree by construction about what "vulnerable" means. Findings go to one rolling issue keyed by the `security-audit` label. This activates the reserve move ADR [0007](0007-nuget-audit-warnings-not-errors.md) names as the correcting move if warnings stop being read; decided while closing issue [#50](https://github.com/smadam813/mimir/issues/50). The decision this ADR canonizes: **when the scheduled run's restore raises `NU1900` (advisory data could not be fetched), the job retries once and then fails the run** — a deliberate inversion of 0007's fail-soft carve-out for the same code.

**Why the inversion is safe here and not in the build**: 0007 keeps `NU1900` fail-soft in the build because a hard failure lands on whoever happened to push next — a time-dependent redden on an innocent commit. A scheduled run has no innocent commit; a red run is nobody's PR and costs nobody's green. What the cron job *cannot* afford is the opposite failure: reading a blind run as clean. That is the same silent-blindness failure mode the job exists to escape, one level up — a workflow that silently reports "all clear" while fetching no advisory data at all is worse than no workflow, because it actively retires the suspicion that would otherwise make someone read the build warnings.

**Blind is indistinguishable from clean — verified empirically, not assumed**: against the .NET 10 SDK, a *reachable* package source that provides no vulnerability data makes `dotnet list package --vulnerable --format json` emit output byte-identical in shape to a genuinely clean run — projects listed with no `frameworks` key, no warning, no `problems` array, exit code `0` — even with a package carrying a known High advisory (Newtonsoft.Json 12.0.1) in the graph. The exit code is not a found/none signal either: the command exits `0` whether or not it finds vulnerabilities. (A fully *unreachable* source does exit non-zero and fails the step on its own.) The only reliable blindness signal is `NU1900` in the restore that runs moments earlier against the same source, which is why the gate lives there. For the same reason the workflow self-tests its JSON parser against committed fixtures before every run: a parser broken by output-shape drift would fail silent-negative forever, and the scheduled run is its only production execution.

**Detection only, one rolling issue, never auto-closed**: the job reports; it does not bump packages or open remediation PRs. On findings it updates the open `security-audit`-labeled issue in place (or opens one if none is open); on a clean run it does nothing — a human closes the issue once the advisories have been acted on, and a later finding after that opens a fresh one. Auto-close was rejected because it would retire the signal on the first clean run after an advisory is withdrawn or a package silently bumped, before anyone confirmed the graph was actually acted on. If dependency-update automation is ever added, the `Npgsql.EntityFrameworkCore.PostgreSQL` / `Microsoft.EntityFrameworkCore.Design` lockstep documented in `Directory.Packages.props` and carried forward by 0007 applies to it unchanged.

**What this ADR does not change**: the build. 0007's carve-out — advisories warn loudly on every restore and never fail a commit — stands exactly as written; this job restores an *active* signal alongside it without re-coupling advisories to unrelated commits.
